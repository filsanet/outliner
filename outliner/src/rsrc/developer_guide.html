CVS Update and Commit Result Codes
?
The file is unknown to CVS
A
The file's been added to your world, but needs to be committed to the repository.
C
There's a conflict.
M
The local copy is different from what's in the repository
You need to diff the files and merge as necessary
P
CVS has sent a patch for the file in order to have it match what's in the Repository.
R
The file's been removed from your private copy, but won't be removed from the repository until you commit the removal
U
The file's been Updated to match what's in the Repository.

Adding a File Format
About this How-To
This How-To shows what you need to know to add a file format for opening and saving documents with JOE.
Note: the code shown here is not really optimal, I'm noticing things that could be cleaned up as I write this, but since the current code works I'll leave it alone for now. [MD]
Editing the file_formats.txt file
Located at: outliner/prefs/file_formats.txt, this file makes JOE aware of which file formats are available to it.
Each line in this file consists of four arguments seperated by the | character. Typically these lines come in pairs: one line for save and one for open.
For example:
file_format|save|com.organic.maynard.outliner.SimpleFileFormat|Plaintext
file_format|open|com.organic.maynard.outliner.SimpleFileFormat|Plaintext
file_format|save|com.organic.maynard.outliner.OPMLFileFormat|OPML
file_format|open|com.organic.maynard.outliner.OPMLFileFormat|OPML
The four arguments
The first argument will always be file_format, indicating that this is a file format definition.
The second argument will be either save or open for save and open file formats respectively.
The third argument is the java class that implements the OpenFileFormat interface or SaveFileFormat interface you want to include.
The fourth argument is the name that user's will see when selecting a file format. This name should be unique, but I don't think uniqueness is enforced at the moment.
So, if you create a new file format of class my.package.FileFormat that implements both the open and save interfaces, you would add lines like this to the file:
file_format|save|my.package.FileFormat|My Format
file_format|open|my.package.FileFormat|My Format
Special characters and blank lines
You can use \ to escape the | char if you need to. Blank lines and lines beginning with the # character are also allowed.
Future: eventually these entries will likely have a file extension argument as well.
Implementing Three Interfaces
You need to implement three interfaces to read and write file formats within JOE.
Implementing the FileFormat Interface
the Extensions methods
public void addExtension(String ext, boolean isDefault);
public void removeExtension(String ext);
public Iterator getExtensions();
public String getDefaultExtension();
public boolean extensionExists(String ext);
Implementing the OpenFileFormat Interface
the Open method
public int open(TreeContext tree, DocumentInfo docInfo);
Your main task in the open method is to construct and add children to the root node of the tree passed in, or to create your own root node and swap it out with the one in the tree.
A basic starting implementation might look something like this:
public int open(TreeContext tree, DocumentInfo docInfo) {
int success = OpenFileFormat.FAILURE;
String text = FileFormatManager.loadFile(docInfo.getPath(), docInfo.getEncodingType());
if (text != null) {
Node newRootNode = new NodeImpl(tree,"");
// Implement code to add child nodes with the data from your file.
if (your code succeeded) {
tree.setRootNode(newRootNode);
success = OpenFileFormat.SUCCESS;
}
}
return success;
}
The code above won't work with remote file systems, or set expanded state of nodes, commented nodes, etc. You should look at the OPMLFileFormat class for an example of how to make a more full featured file format. Most of those features involve setting values on the DocumentInfo object.
Implementing the SaveFileFormat Interface
the Save method
public boolean save(TreeContext tree, DocumentInfo docInfo)
Your main task in the save method is to save your file and return success for failure.
Here's the outline of a basic implementation:
public boolean save(TreeContext tree, DocumentInfo docInfo) {
// Implement code to turn the tree into the format you want to save
if (your code succeded) {
return true;
}
else {
return false;
}
}
the supportsXxxxx Methods
boolean supportsComments();
This method should return true if the file format supports comments, false if not.
This lets JOE warn a user who is about save a commented outline to a format that doesn't store comments.
boolean supportsEditability();
boolean supportsMoveability();
boolean supportsAttributes();
boolean supportsDocumentAttributes();

Working with gui_tree.en.xml
String Constants
Make an asset entry in gui_tree.en.xml that's a child of the <!-- TEXT REPOSITORY --> node
example
<asset type="text" key="trunc_string" value=" . . "/>
To obtain the value, use the GUITreeLoader.reg.getText method
example
String TRUNC_STRING = GUITreeLoader.reg.getText("trunc_sring");
Number Constants
Preferences
Make an entry in Preferences.java that defines a string constant for the asset entry
Propagating gui_tree.en.xml
When you make changes to gui_tree.en.xml, copy it to the other language versions and cvs commit those as well
That lets the translators discover they've got work to do
